# 全体モード 追加チェック群

全体モードでは `checks/schema.md` のチェックに**加えて**、このファイルのチェックを実行する。
対象は、ユーザーが指定したディレクトリ配下のクエリ・モデル・リポジトリ・マイグレーション。

> このファイルにも改善案は載せていない。改善案は深掘りフローで、実コードを読んだ上で生成する。

## 構成

- **Part A — 全体モード専用チェック**: クエリやビューを見ないと検出できないアンチパターン。
- **Part B — スキーマチェックの全体モード補強**: `schema.md` の各チェックを、クエリ検査で確証に引き上げる。

---

## Part A — 全体モード専用チェック

### やり過ぎたJOIN

**概要**: JOINの多用・濫用でクエリが複雑化・低速化する。過剰な結合数、不要なJOIN、N+1問題を含む。

**深刻度**: HIGH

**検出方法**:

- 1クエリあたりのJOIN本数が過剰（**5本以上**を閾値の目安に）。
- N+1パターン: ループ内でのクエリ発行、ORMで `include`/`preload`/`with` を伴わない関連アクセス、`.map` 内での `find`/`findUnique`。
- 自己結合・結合条件の欠落で直積になりうるJOIN。

**許容されるケース**: 集計・分析用の重いJOINは意図的なこともある。バッチ/非同期実行で許容範囲に収まっているか。頻出する複雑JOINはView化で参照側を簡素化できる。

---

### ソートの依存

**概要**: `ORDER BY` + `OFFSET/LIMIT` による深いページング。OFFSETが大きいほど毎回ソート・スキャンが走り、ページが進むほど線形に遅くなる。

**深刻度**: HIGH

**検出方法**:

- 変数OFFSETを伴うクエリ / ORM呼び出し（`OFFSET ?`, `.offset()`, `.skip()`, `LIMIT ? OFFSET ?`）。
- 一覧系エンドポイントで大きなOFFSETが想定されるページネーション。
- ORDER BY列にINDEX（カバリング含む）があるかをスキーマと突合。

**許容されるケース**: 管理画面など件数が少なく浅いページしか辿らないなら許容。深いページングが想定される箇所はキーセットページネーション（seek法）への置換候補。

---

### 知らないロック

**概要**: ロックの種類（排他/共有）と粒度（表/行）を理解せず、想定外のロック競合やデッドロックを引き起こす。RDBMSごとの挙動差（MySQL InnoDBのギャップロック等）が根本原因。

**深刻度**: MEDIUM（影響が出ると HIGH 相当だが、静的検出の確度は低〜中）

**検出方法**:

- 同一トランザクション内で複数テーブルにロックを取る際の、テーブル/行のロック獲得順序が箇所によって不一致。
- DDL（ALTER等）と参照クエリが同じテーブルに競合。
- MySQLでギャップロック/ネクストキーロックを誘発しうる範囲DELETE/UPDATE（`WHERE` が範囲条件で非INDEX列）。

**許容されるケース**: 静的解析では断定しづらいため、必ず「可能性」として提示する。実際の競合はトランザクション境界とアクセス順序の確認が必要。

---

### ロックの功罪

**概要**: トランザクション分離レベルを理解せず安易に変更し、Lost Update や Phantom Read を招く。

**深刻度**: HIGH

**検出方法**:

- 分離レベルの変更箇所（`SET TRANSACTION ISOLATION LEVEL ...`、ORMの分離レベル指定）。
- read-modify-write 型の更新（SELECTした値を計算してUPDATE）が、`SELECT ... FOR UPDATE` や楽観ロック（バージョン列）なしで行われている。

**許容されるケース**: 分離レベルを下げる明確な理由があり、不整合リスクを評価済みなら許容。RDBMS既定の違い（MySQL: repeatable read / PostgreSQL: read committed）を踏まえているか確認する。

---

### 複雑なクエリ（スパゲッティクエリ）

**概要**: 1本のSQLに集計・結合・サブクエリを詰め込むスパゲッティクエリ。可読性・保守性・性能が悪化する。

**深刻度**: MEDIUM

**検出方法**:

- サブクエリのネスト階層が深い（**3階層以上**を目安に）。
- 相関サブクエリ・スカラサブクエリの多用。
- JOIN数とサブクエリ数を合わせたクエリ複雑度が突出している箇所。

**許容されるケース**: 性能上の理由で意図的に1本にまとめている場合はトレードオフを明示できているか。複数クエリ分割・中間テーブル化・CTEでの可読性改善が候補。

---

### ゾンビマート・多段マート

**概要**: BI/DWH領域の2大アンチパターン。**ゾンビマート**=設計書がなく不要になっても消せず幽霊化したデータマート。**多段マート**=マートからさらにマートを作り、データ鮮度と導出経路が不透明になる。

**深刻度**: MEDIUM

**検出方法**:

- マート → マートの依存チェーン（あるマートのソースが別のマートになっている）。
- 定義が類似したマートの重複。
- 長期間アクセスされていない / 設計書のないマートテーブル（メタデータがあれば）。

**許容されるケース**: セントラルDWHから切り出した1段目のマートで設計書が整備されている場合は正当。マテリアライズドビューとして更新タイミングまで管理されていれば健全。

---

### 多段ビュー

**概要**: ビューの上にビューを重ねる設計。ビューアクセスは「ユーザーのSELECT + ビュー定義のSELECT」が合成されるため、段数が増えるほど性能悪化と依存の不透明化を招く。

**深刻度**: MEDIUM

**検出方法**:

- ビュー定義SQL（`CREATE VIEW`）が別のビューを参照している。
- ビューの依存関係が**3段以上**。
- 同じ機能のビューが乱立している。

**許容されるケース**: マテリアライズドビューで多段を実現している場合は性能問題が出にくい（ただし更新タイミングの設計が必要）。原則はビュー1段で基底テーブルを直接参照。

---

## Part B — スキーマチェックの全体モード補強

`schema.md` のチェックは、クエリを見ることで「示唆」から「確証」に引き上げられる。
スキーマ側で該当した項目について、対応する全体モードの検出を追加で行う。

| schema.md のチェック | 全体モードで追加する検出 | 効果 |
|---|---|---|
| 命名・カラム意味の明確性 | アプリコード/クエリで参照ゼロのテーブル・カラム（死蔵）。同一カラムが分岐で別意味に使われる箇所 | 死蔵カラムの確証、ダブルミーニングの検出 |
| 失われた事実 | 金額・ステータスカラムへの UPDATE/DELETE 発行 | 履歴喪失の確証（CRITICAL に引き上げ） |
| フラグの闇 | 全SELECTに `WHERE ... = 0`（削除フラグ除外）が常駐している | フラグの波及範囲を可視化 |
| EAV・ポリモーフィック・STI | IDの `substr()`/正規表現分解による条件分岐（スマートカラム）。同一テーブルの複数目的流用 | 隠された状態の確証 |
| 非スカラ値・ジェイウォーク | サンプル/シードデータのカンマ区切り値。`FIND_IN_SET`/`string_to_array`/`LIKE '%,%'` での値分解 | 第1正規形違反の確証 |
| テーブル分割 | `UNION ALL` で同型テーブルを結合するクエリ | 水平分割の確証 |
| 不適切なキー | 結合クエリでの型変換（`CAST`/暗黙変換） | 型不一致結合の確証 |
| ダブルマスタ | `FULL OUTER JOIN` / `UNION` + `COALESCE` で複数マスタを束ねるクエリ | ダブルマスタの確証 |
| アドホックな集計キー | GROUP BY句の CASE 式による分類（代替手段の使用有無） | 集計キー後付けの是非 |
| データクレンジング | 同一テーブル内の類似文字列（表記揺れ兆候） | 名寄せ未実施の確証 |
| 効かないINDEX | 複合INDEXの列順 vs クエリの WHERE/ORDER BY 条件の不一致。後方一致LIKE（`LIKE '%xxx'`）でINDEXが効かない箇所 | INDEX不発の確証 |
| JSONの甘い罠 | JSON演算子（`->`, `->>`, `@>`, `jsonb_array_elements`）でJSON内部を頻繁に WHERE/JOIN/集計 | 「罠」の確証（HIGH に引き上げ） |
