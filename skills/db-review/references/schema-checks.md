# スキーマモード チェック群

スキーマファイル（DDL / マイグレーション / モデル定義）から検出できるアンチパターンのチェック集。
各チェックの `検出方法` に従って走査し、該当箇所を集める。`許容されるケース` は誤検出・深刻度判断の材料に使う。

> このファイルには改善案は載せていない。改善案は深掘りフローで、実コードを読んだ上で生成する。

> **スキーマモードの断定回避（重要）**: 「UNIQUE/FKの欠如」「履歴テーブルの有無」は、追記ログ設計（最新行＝現在）・UPSERT運用・アプリ層担保かもしれず、スキーマだけでは欠陥と断定できない。これらは「欠落の可能性（要コード確認）」と仮説で書き、結果（「複数行が作れてしまう」等）を事実として断定しない。

### 命名・カラム意味の明確性

**概要**: 意味のわからない命名・正体不明のカラムが放置され、DBが「迷宮」化する。スペルミス・汎用名・コメント欠落が代表例。

**深刻度**: MEDIUM

**検出方法**:

- 典型スペルミス辞書との照合: `falg`, `flg`, `adress`, `addres`, `regist(a)r`, `recieve`, `lenght`, `defualt` 等。
- 汎用語の濫用: `data`, `info`, `detail`, `flag`, `kbn`, `区分`, `type`, `value`, `tmp`, `column1`, `col1` 等が単独カラム名になっている。
- カラムコメント（`COMMENT ON` / Prismaの `///` / Railsの `comment:`）が無く、かつ型が汎用的（`int` / `varchar` / `text`）。
- 意味不明な略語・ローマ字綴り（`syurui`, `tnpo`, `flg2` 等）。

**許容されるケース**: スペルミス命名は既存アプリが依存しており即時改名コストが高いことがある（互換ビュー/別名で吸収しつつ移行する選択肢がある）。業界慣習の略語（`amt`, `qty`, `dt`）は許容。

### 失われた事実

**概要**: UPDATE上書き・物理DELETEで過去の「事実（履歴）」が失われる。料金改定・ステータス変更・契約変更で「当時の値」を復元できなくなる。

**深刻度**: CRITICAL

**検出方法**:

- 金額・ステータス・契約・料金系カラム（`price`, `amount`, `fee`, `status`, `state`, `plan`, `rate` 等）を持つテーブルに、対応する履歴テーブル（`*_history`, `*_logs`, `*_archive`, `*_revisions`）が**存在しない**。
- 同テーブルに有効期間カラム（`valid_from`/`valid_to`, `effective_date`, `expired_at`, `start_date`/`end_date`）が無い。
- これは「弱い示唆」。確証は全体モードで金額・ステータスカラムへのUPDATE/DELETE発行を検出して得る。

**許容されるケース**: 履歴が業務上不要なデータ（一時データ、キャッシュ、再生成可能なデータ、ログの揮発許容）なら上書きで問題ない。また「最新行＝現在」の追記ログ設計（`ORDER BY created_at DESC LIMIT 1` で現在値を引く等）なら、UNIQUE欠如・複数行は欠陥ではなく設計通り。検出は「候補提示」に留め、UNIQUE/複数行をもって即欠陥と断定しない。

### フラグの闇

**概要**: テーブルに「状態」を持たせるアンチパターン。代表が削除フラグ。全クエリに `WHERE delete_flag=0` が波及し、UNIQUE制約・FK・INDEXが効かなくなる。

**深刻度**: CRITICAL（論理削除フラグ）／状態カラム + 制約欠如は HIGH

**検出方法**:

- 論理削除フラグ: `delete_flag`, `del_flag`, `is_deleted`, `deleted`, `deleted_at`, `removed`, `disabled` 等のカラム。
- 状態カラム: `status`, `state`, `kbn`, `区分`, `phase` 等。boolean / 低カーディナリティ（取りうる値が数個）。
- これらの状態カラムを持つテーブルで、UNIQUE制約・外部キーが未設定（フラグのせいで一意制約が張れていない兆候）。

**許容されるケース**（書籍が許容条件を明記）: ①テーブルが小さくINDEXが不要 ②そのテーブルが親にならず頻繁なJOIN対象でない ③UNIQUE制約が不要でFKで整合性を担保する必要がない。`deleted_at` 方式（ソフトデリート）自体は広く使われ、上記条件を満たせば許容範囲。

### 参照アクション（onDelete）とカラム制約の不整合

**概要**: 外部キーの削除時アクション（onDelete）とカラムの制約が噛み合わず、意図と実挙動がズレる。代表は `set null` を NOT NULL 列に設定するケースで、親削除時に NOT NULL 違反で失敗し、実質 restrict として振る舞う（しかもエラーがFK由来だと気づきにくい）。

**深刻度**: CRITICAL（set null × NOT NULL の矛盾）／意図しないcascade・アクション不統一は HIGH

**検出方法**:

- **set null × NOT NULL**: FKが `onDelete: set null`（Drizzle `{ onDelete: 'set null' }` / Prisma `onDelete: SetNull` / Rails `on_delete: :nullify` / TypeORM `'SET NULL'`）なのに、そのFK列が NOT NULL（Drizzle `.notNull()` / Prisma 非optional / `null: false`）。→ 親削除がNOT NULL違反で失敗する。
- **意図しない cascade**: `created_by` / `updated_by` / `owner_id` など「作成者・所有者」参照に `onDelete: cascade` が付いている（作成者を消すと本体まで消える）。要レビュー。
- **同役割カラムでアクション不統一**: `created_by` 等の同じ役割の列が、テーブルごとに set null / cascade / restrict / 無指定（NO ACTION）とバラバラ。削除方針が定まっていない兆候。
- **FKに削除アクション未指定**: 親削除時の挙動が暗黙（NO ACTION≒restrict）。意図的かを確認。

**許容されるケース**: 親（ユーザー等）を物理削除しない運用（論理削除・匿名化）なら set null × NOT NULL の矛盾は顕在化しない。ただし「set null と書いてあるのに set null されない」誤読リスクは残るため、nullable化かコメントで意図を一致させると安全。cascade も「親が消えたら子も消すのが正しい」エンティティ（明細・添付など）なら妥当。

### EAV・ポリモーフィック・STI

**概要**: カラムやIDに「状態・意味」を暗黙的に埋め込む設計。FK制約・型・NOT NULLの恩恵を失う。

**深刻度**: HIGH（ポリモーフィック関連のFK欠如はデータ不整合リスクで CRITICAL 寄り）

**検出方法**:

- **EAV**: `entity_id` + `attribute`(or `key`/`name`) + `value` の縦持ち3列テーブル。`*_attributes`, `*_meta`, `*_options`, `properties` 等の命名。
- **ポリモーフィック関連**: `xxx_type` + `xxx_id` のペア列があり、FK制約が張られていない（Rails: `commentable_type`/`commentable_id`、`*able_type`/`*able_id`）。
- **STI**: `type` カラム + サブタイプ別のnullable列が多数。種別ごとに常にNULLになる列が混在し、それらに NOT NULL 制約が無い。

**許容されるケース**: 属性が動的に増え続ける等、真にスキーマレスが必要なケースは限定的に許容（ただしドキュメント必須）。STIは開発速度のためにビジネス判断で採用することがある（ただし「DBの寿命はアプリより長い」点に留意）。

### 非スカラ値・ジェイウォーク

**概要**: 1つのセルに複数の値を入れる設計。カンマ区切り値（ジェイウォーク）と配列型が代表例。第1正規形に反し、FK・型・INDEXの恩恵を失う。

**深刻度**: HIGH

**検出方法**:

- カンマ区切りを格納する疑いのあるカラム: `varchar`/`text` 型で名前が `_ids`, `_list`, `tags`, `categories`, `codes` 等の複数形 / リスト系。
- 配列型宣言: DDLの `varchar[]`, `integer[]`, `text[]`、Prismaの `String[]`（リレーションでないスカラ配列）。
- INSERT/シードデータに `'1,2,3'` や `'{a,b,c}'` のような複数値リテラル。

**許容されるケース**: 検索・集計の対象にならず表示用途のみなら限定的に許容。性能上の理由でPostgreSQL配列型/JSONBを使う場合はトレードオフを明示できているか。意味を壊す過剰分割（メールアドレスの `@` 以下分割など）は逆にしない。

### 繰り返しカラム

**概要**: 同種の意味を持つカラムが連番で横に並ぶ設計（マルチカラムアトリビュート / 列持ちテーブル）。`tel1, tel2` や `同伴者1, 同伴者2, 同伴者3` が代表例。要素追加に ALTER TABLE が必要。

**深刻度**: HIGH

**検出方法**:

- カラム名の連番サフィックス: 正規表現 `^(.+?)[_]?\d+$` にマッチする同型カラムが同一テーブルに**3つ以上**（`item_1/2/3`, `child1/2/3`, `phone_1/2`, `address_2`）。
- 同型・同義カラムの水平展開。多くがnullableで埋まる想定のもの。
- `その他` / `etc` / `備考2` のような正規化逃れカラム。

**許容されるケース**: 繰り返し回数が業務的に確定（四半期=必ず4列など）、帳票フォーマットに合わせる必要がある、パフォーマンス要件で行持ちから意図的に変換した結果、のいずれか。**誤検出注意**: 履歴データ（イベントソーシング・追記型）は正当な設計。繰り返し・重複を機械的に不整合と断定しない。

### 強過ぎる制約

**概要**: CHECK制約やDOMAIN型にビジネスルールを盛り込み過ぎる。仕様変更時に ALTER（ACCESS EXCLUSIVEロック）が必要で変更に弱い。

**深刻度**: MEDIUM

**検出方法**:

- `CREATE DOMAIN` での複雑な正規表現チェック。
- CHECK制約に変わりやすいビジネスルール（金額の上限、ステータスの組み合わせ等）が埋め込まれている。
- ENUM型で値を固定している（マスタ化すべき選択肢）。
- 時間・状態依存のCHECK（`updated_at >= now()` 等）。

**許容されるケース**: NOT NULL・主キー・参照整合性は「データを守る最後の砦」として必ずDB側に置くべきで、これらは検出対象外（むしろ推奨）。普遍的で変わらない制約はDBに残してよい。

### 単一参照テーブル（OTLT）

**概要**: 本来別々であるべき複数のマスタ（コード類）を「コードタイプ＋コード値＋コード内容」の1テーブルに寄せ集める設計。FK制約が張れず副作用が大きい。

**深刻度**: HIGH

**検出方法**:

- 汎用3列構成のマスタ: `code_type`(or `type`/`category`) + `code_value`(or `code`/`value`) + `code_name`(or `name`/`label`) のテーブル。`m_code`, `codes`, `common_master`, `general_master` 等の命名。
- コード値カラムが大きめの可変長（`varchar(255)` 等）。
- このテーブルへ向けた外部キー制約が設定されていない（できない）。

**許容されるケース**: UIドロップダウン表示のみで参照整合性が不要、またはマスタ種別が極端に多く各々が数件程度で運用簡素化のメリットが上回る場合は限定許容。

### テーブル分割

**概要**: パフォーマンス目的で1テーブルをレコード単位（水平）または列単位（垂直）で物理分割する設計。正規化理論からは導けない純粋な物理要請で原則避ける。

**深刻度**: MEDIUM

**検出方法**:

- **水平分割**: テーブル名に年度・月・地域などの値が含まれる（`sales_2024`, `sales_2025`, `orders_tokyo`, `log_202406`）。
- **垂直分割**: 同一の主キーを持つ別テーブルに列が分散し、常にJOINで再結合が必要。

**許容されるケース**: シャーディングが必要な規模で意図的に設計している場合は許容。カラムベースDB（BI/DWH用途）の垂直分割も正当。本来はDBMSのパーティション機能（RANGE/LIST/HASH）で代替できないかを確認する。

### 不適切なキー

**概要**: キー列（主キー・外部キー・結合キー）に不適切なデータ型を使う。代表は可変長文字列（VARCHAR）の「名前」をキーにすること。不変性が無く結合がアンマッチになる。

**深刻度**: HIGH

**検出方法**:

- 主キー/外部キーが VARCHAR で、内容が名称系（`company_name`, `user_name`, `product_name` 等の変動値）。
- FK参照元と参照先でキーカラムのデータ型が不一致（`CHAR` vs `VARCHAR`、文字列 vs 数値、桁数違い）。
- ゼロ埋め有無の不統一が疑われる（`code char(3)` と `code int` の混在）。

**許容されるケース**: ISO国コード（`JP`）・通貨コード（`JPY`）など国際標準で不変性が保証されたコードは文字列キーでも問題ない。

---

### ダブルマスタ

**概要**: 同じ役割のマスタテーブルが2つ存在する。通常設計では起きないが、システム統合時に別システムのマスタ同士が同居して発生する。

**深刻度**: HIGH

**検出方法**:

- 同じエンティティを表すマスタが複数（`customers_a`/`customers_b`, `users_legacy`/`users_new`, `m_item`/`item_master`）。
- 列定義が酷似したテーブルが複数存在する。
- 確証は全体モードで FULL OUTER JOIN / UNION + COALESCE のクエリを検出して得る。

**許容されるケース**: システム統合の移行期間中で、段階的に統合を進めている場合は期限付きで許容。

---

### 代理キー

**概要**: 主キーを決められないときに自然キーの代わりに追加する人工連番キー。基本スタンスは「可能な限り自然キーを使い、代理キーは避ける」。グレーノウハウ。

**深刻度**: MEDIUM（自然キーに UNIQUE 未設定で重複が起きうる場合は HIGH）

**検出方法**:

- 連番PK（`id` / `seq` / `surrogate_id`、AUTO_INCREMENT / SERIAL / IDENTITY）があり、かつ自然キー候補（業務コード等）に UNIQUE 制約が**未設定**。
- 複合自然キーが2列以下で十分一意なのに、わざわざ代理キーを追加している。

**許容されるケース**: コードの使い回し・対象の変化がある業務、自然キーが存在しないデータ、複合自然キーが3列以上で結合の実用性が低い場合は代理キーが有効。代理キーを使う場合でも自然キーへの UNIQUE 制約があれば健全。

---

### アドホックな集計キー

**概要**: BI/DWH文脈で、既存にない粒度で集計したいとき、元テーブルに集計用コード列を安直に追加してしまう設計。グレーノウハウ。

**深刻度**: MEDIUM

**検出方法**:

- マスタテーブルに集計・分類目的だけのコード列が後付けされている（例: 都道府県テーブルに `region_code`, `area_div`）。命名ヒューリスティックで検出。
- 大きなテーブルに集計用の列が複数追加されて肥大化している。

**許容されるケース**: コード体系が安定し長期間変更されない、またはテーブルが小さく性能影響が無視できる場合は許容。本来は変換テーブル分離 / ビュー / GROUP BY内CASEで代替できる。

### データクレンジング（名寄せ）

**概要**: 業務データをDB登録可能な「きれいな」状態にする洗浄作業。名寄せをサボるとダブルマスタを、一意キー不在は不適切なキーのアンチパターンを生む。

**深刻度**: MEDIUM

**検出方法**:

- 一意識別子（顧客コード等）が無いテーブルに、フリーハンド入力のテキスト列（氏名・会社名・住所）がある（PKなし + 自由入力テキストの組み合わせ）。
- 同種マスタが複数存在する（ダブルマスタの兆候）。
- 表記揺れの確証は全体モード/サンプルデータで得る（`渡邊`/`渡辺`、`㈱`/`株式会社`）。

**許容されるケース**: 新規システムで最初からコード体系を設計している場合はクレンジング不要。蓄積済みデータやシステム統合ではほぼ例外なく必要。

### 効かないINDEX

**概要**: INDEXを張っているのに効かない／効かせられない。低カーディナリティ列へのINDEX、インデックスショットガン（無計画な大量INDEX）が代表例。

**深刻度**: HIGH

**検出方法**（スキーマから見える分）:

- 低カーディナリティ列（boolean, `gender`, `status`, `type`, `flag` 等）への**単独**INDEX。
- INDEX過多のテーブル（1テーブルにINDEXが多数 ＝ インデックスショットガン）。
- 全体モードでは、複合INDEXの列順 vs クエリ条件の不一致、後方一致LIKEとの突合を追加で行う（`checks/full.md` 参照）。

**許容されるケース**: 書き込み重視のテーブルはINDEXを敢えて絞ることがある。INDEX過多の検出時も、そのテーブルが読み取り中心か更新中心かを加味する。

---

### JSONの甘い罠

**概要**: リレーショナルに構造化すべきデータを何でもjson/jsonb列に詰め込む。FK制約が効かず、型・NOT NULL・INDEXの恩恵も失う。

**深刻度**: MEDIUM（スキーマ上は「存在検出」のため示唆。全体モードでJSON内部の検索/結合を検出すると HIGH に確証が上がる）

**検出方法**（スキーマから見える分）:

- DDL / モデルで `json` / `jsonb` 列、Prismaの `Json` 型を検出。列名・テーブルの役割から「構造化すべきデータ」かを推測。
- 全体モードでは JSON演算子（`->`, `->>`, `@>`, `jsonb_array_elements`）でJSON内部を頻繁に WHERE/JOIN/集計しているかを確認する（`checks/full.md` 参照）。

**許容されるケース**: 外部APIレスポンス、プラグイン設定、ログ的データなど、真に非構造でスキーマが頻繁に変わるものはJSONが適切。「検索キーにするか」「整合性が要るか」で判断する。
